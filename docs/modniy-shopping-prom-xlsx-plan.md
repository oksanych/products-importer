# Перевірений План MVP: Modniy Shopping → Prom XLSX

**Summary**
- MVP створює XLSX для першого імпорту нових retail-only товарів з Modniy Shopping у Prom.
- Товари створюються як різновиди за українським розміром.
- План звірено з наданим експортом XLSX, 5 URL товарів і документацією Prom.
- Головні правила: `ID_групи_різновидів` має бути числовим; різновиди мають відрізнятися не більше ніж за 3 характеристиками; для MVP змінюється тільки `Розмір жіночого одягу (UA)`.
- `Код_товару` формується зі службовим префіксом `MS` і ручним двозначним ідентифікатором кольору.

**Confirmed Prom Rules**
- Використовувати структуру XLSX з наданого експорту: листи `Export Products Sheet` і `Export Groups Sheet`, 128 колонок, 25 блоків характеристик.
- Для XLSX Prom очікує дві вкладки: `Export Products Sheet` і `Export Groups Sheet`.
- Кожен рядок `Export Products Sheet` описує одну товарну позицію або послугу.
- `Унікальний_ідентифікатор`:
  - для першого створення нових товарів лишати порожнім;
  - це службовий позитивний числовий ідентифікатор Prom;
  - він автоматично заповнюється Prom при експорті;
  - його не треба вигадувати або змінювати вручну;
  - для подальших оновлень уже імпортованих товарів потрібно використовувати значення з експорту Prom, інакше повторний XLSX може створити дублікати.
- `Ідентифікатор_товару`:
  - обов'язковий;
  - формат у документації: рядок до 255 символів;
  - може містити символи та числа;
  - значення на кшталт `modniy-3064637917-48` допустиме. Знак `-` не названий окремо саме для цього поля, але він підпадає під "символи"; це обґрунтований висновок із формату поля.
- `ID_групи_різновидів`:
  - має бути числом від `0` до `999999999`;
  - усі рядки одного товару-різновиду мають мати однаковий `ID_групи_різновидів`;
  - використовувати `crc32(product_id) % 900000000 + 1`;
  - перевіряти колізії у файлі.
- Різновиди мають бути створені на основі характеристик основного товару.
- Для MVP використовувати одну змінну характеристику: `Розмір жіночого одягу (UA)`.
- Не використовувати більше 3 змінних характеристик.
- Деталі таблиці розмірів (`intl_size`, бірка, стегна, чашка) не робити окремими змінними характеристиками; показувати їх в описі.
- `Тип_товару = r`.
- `Оптова_ціна` і `Мінімальне_замовлення_опт` завжди порожні.
- `Наявність = +`, `Кількість = 3` для кожного розміру.
- `Супутні`, `ID_Супутніх`, `Подарунки`, `ID_Подарунків` лишати порожніми для MVP.

**Identifier Strategy**
- Для першого імпорту нових товарів:
  - `Унікальний_ідентифікатор = ""`
  - `Ідентифікатор_товару = modniy-{product_id}-{ua_size}`
- Для майбутнього режиму оновлення товарів:
  - не генерувати новий `Унікальний_ідентифікатор`;
  - брати його з актуального експорту Prom;
  - зіставляти товар за стабільним `Ідентифікатор_товару`;
  - оновлювати рядок, не створюючи дублікат.
- `Ідентифікатор_товару` має бути стабільним: якщо товар `3064637917`, розмір `48`, то значення завжди `modniy-3064637917-48`.
- Нове правило для `Код_товару` не змінює `Ідентифікатор_товару`.

**Color ID And Product Code Strategy**
- Під час запуску скрипта обов'язково передавати `--color-id`.
- `--color-id` має бути string рівно з 2 цифр: `^\d{2}$`.
- Значення з початковим нулем допустимі й мають зберігатися як string, наприклад `05`.
- Якщо `--color-id` відсутній, переданий без значення або має неправильний формат, скрипт має завершитися до генерації XLSX.
- У терміналі при такій помилці має бути текст `Check color parameter`.
- `Код_товару` формується так:
  - взяти `sku` з JSON-LD `Product`;
  - обрізати тільки кінцевий суфікс формату `-\d+`, якщо він є;
  - додати префікс `MS`;
  - додати наприкінці `color_id`.
- Приклади:
  - `3903` + `65` -> `MS390365`
  - `58800-25` + `45` -> `MS5880045`
  - `58800-25` + `05` -> `MS5880005`
- Однаковий `Код_товару` для всіх розмірів одного товару/кольору є очікуваною поведінкою; унікальність різновидів забезпечує `Ідентифікатор_товару`.

**Import Price Strategy**
- Під час запуску скрипта обов'язково передавати `--import-price`.
- `--import-price` має бути цілою сумою в UAH, наприклад `1000`.
- Значення `1000.00` і `1000,00` допустимі та нормалізуються до `1000`.
- Якщо `--import-price` відсутній, переданий без значення або має неправильний формат, скрипт має завершитися до генерації XLSX.
- У терміналі при такій помилці має бути текст `Check price parameter`.
- Значення записується в колонку `Ціна` для всіх рядків різновидів.

**Parser**
- Брати `name`, `sku`, `brand`, `price`, `currency`, `availability`, `description` з JSON-LD `Product`.
- Брати характеристики з `table[data-qaid="characteristics_block"]`.
- Брати фото з `og:image`, JSON-LD `image` і gallery HTML.
- Видаляти дублікати фото.
- Парсити таблиці розмірів у горизонтальному й вертикальному форматах.
- Якщо таблиці немає, витягувати числові розміри з назви перед словом `розмір`.
- Ідентифікатор кольору не парсити зі сторінки; він завжди задається вручну через CLI.

**CLI**
- Базовий запуск:

```bash
python3 generate_import.py "https://modniy-shopping.com.ua/ua/p3064637917-slitnyj-kupalnik-fuba.html" --color-id 65 --import-price 1000
```

- Запуск із локального HTML fixture:

```bash
python3 generate_import.py "https://modniy-shopping.com.ua/ua/p3064637917-slitnyj-kupalnik-fuba.html" --from-html fixtures/modniy_3064637917.html --color-id 65 --import-price 1000
```

- Помилки `--color-id`:
  - без `--color-id` -> exit code `2`, stderr містить `Check color parameter`;
  - `--color-id` без значення -> exit code `2`, stderr містить `Check color parameter`;
  - `--color-id 5`, `--color-id 005`, `--color-id ab` -> exit code `2`, stderr містить `Check color parameter`.
- Помилки `--import-price`:
  - без `--import-price` -> exit code `2`, stderr містить `Check price parameter`;
  - `--import-price` без значення -> exit code `2`, stderr містить `Check price parameter`;
  - `--import-price 0`, `--import-price abc`, `--import-price 999.50` -> exit code `2`, stderr містить `Check price parameter`.

**Row Builder**
- Один рядок на кожен розмір.
- Окремого parent-only рядка не створювати, бо Prom вказує, що основний товар також є одним із різновидів.
- `Код_товару = MS{sku_without_trailing_numeric_suffix}{color_id}`.
- `Ідентифікатор_товару = modniy-{product_id}-{ua_size}`.
- `Унікальний_ідентифікатор = ""` тільки в режимі створення нових товарів.
- `ID_групи_різновидів = numeric_crc32_group_id(product_id)`.
- Група:
  - `Номер_групи = 85929329`
  - `Назва_групи = Купальники`
  - `Посилання_підрозділу = https://prom.ua/Kupalniki-zhenskie`
  - `Ідентифікатор_підрозділу = 31299`
  - `Ідентифікатор_групи = ""`
- Описи, фото, SEO і базові характеристики дублювати в кожному рядку-різновиді.
- Характеристики записувати позиційно в repeated triplets, не через dict з однаковими ключами.
- У кожному рядку групи має бути однаковий набір характеристик.
- Відрізнятися має тільки значення `Розмір жіночого одягу (UA)`.

**Manual Acceptance**
- Зробити пробний імпорт одного XLSX у Prom.
- Перевірити, що Prom створює саме різновиди, а не окремі товари.
- Перевірити, що `Код_товару` у Prom має формат `MS{sku_without_trailing_numeric_suffix}{color_id}`.
- Після успішного імпорту зробити експорт з Prom і перевірити, що Prom присвоїв `Унікальний_ідентифікатор`.
- Зберегти цей експорт як fixture для майбутнього update-mode.
- Якщо імпорт не пройде, зберегти звіт помилок Prom як fixture.

**Assumptions**
- У кабінеті Prom увімкнено застосунок/можливість `Різновиди`; без цього Prom не імпортує різновиди.
- Category mapping `31299 / Жіночі купальники` підтверджено на сторінках джерела й у наданому експорті; окремий список категорій Prom не змінюється в межах MVP.
- CRC32 group ID є технічним правилом MVP, яке задовольняє підтверджене обмеження Prom на числовий `ID_групи_різновидів`.
- `color_id` завжди задається вручну під час запуску скрипта і не парситься зі сторінки товару.
- `color_id` зберігається як string, щоб `05` не перетворювався на `5`.
- Обрізається тільки кінцевий суфікс SKU формату `-числа`; інші частини SKU не змінюються.
